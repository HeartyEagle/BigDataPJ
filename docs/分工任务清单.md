# 分工任务清单

本文档是给组员执行用的任务书。所有人按数据处理流水线推进，先完成最小可运行版本，再做优化和加分项。

## 0. 总体流水线

```text
原始监控数据
  -> 成员 2：Hadoop 环境和数据导入
  -> 成员 3：数据清洗、时间序列构造、统计画像
  -> 成员 4：本地 Baseline 异常检测
  -> 成员 5：Hadoop 分布式异常检测、性能测试
  -> 成员 6：Demo 可视化、PPT、项目文档
  -> 成员 1：接口规范、集成、进度和答辩总控
```

统一性能指标接口：

```text
timestamp, cmdb_id, kpi_name, value
```

统一异常结果接口：

```text
timestamp, cmdb_id, kpi_name, value, method, is_anomaly, score, threshold_low, threshold_high
```

一条时间序列定义为：

```text
cmdb_id + kpi_name
```

## 1. 成员 1：组长 / 架构 / 集成负责人

### 需要实现的功能

1. 维护统一数据接口和结果接口。
2. 维护项目目录结构和 HDFS 路径规范。
3. 维护主流程脚本，保证清洗、统计、算法、Hadoop 输出能接起来。
4. 检查每个成员的输出文件是否能被下游读取。
5. 汇总所有模块的运行方式、截图、结果和答辩材料。

### 预期效果

1. 全组只认一套字段名，不出现各写各的情况。
2. 任意成员交付文件后，下游可以直接读取。
3. 最后能讲清楚从原始数据到异常结果的完整链路。
4. 答辩时可以展示 Hadoop 集群、HDFS 目录、MapReduce 任务和 Demo 页面。

### 需要修改的文件

| 文件 | 作用 |
| --- | --- |
| `README.md` | 项目入口说明，告诉别人项目是什么、怎么运行、主要文档在哪里。 |
| `configs/pipeline.yaml` | 全局路径、字段、Hadoop 路径和算法参数配置。新增公共配置先写这里。 |
| `src/utils/data_contract.py` | Python 代码里的统一字段定义。字段变更必须先改这里。 |
| `src/utils/io.py` | 统一读写 CSV/Parquet 的工具函数，避免每个人重复写读取逻辑。 |
| `scripts/run_all/run_pipeline.ps1` | 主流程脚本，串起清洗、统计、baseline 和 Hadoop 作业。 |
| `docs/组长交付说明.md` | 组长视角的项目规划、接口和交接说明。 |
| `docs/分工任务清单.md` | 给组员执行的详细任务清单。 |

### 验收标准

```text
1. 每个成员知道自己要改哪些文件。
2. 数据字段和结果字段在文档、代码、Demo 中一致。
3. 主流程至少能在小样本上跑通。
4. 组长能指出每个结果文件由谁产生、给谁使用。
```

## 2. 成员 2：Hadoop 集群环境 / 数据导入负责人

### 需要实现的功能

1. 搭建三台服务器 Hadoop 集群。
2. 保证 master 和 worker 之间 SSH 免密。
3. 配置 HDFS、YARN、MapReduce。
4. 创建 HDFS 项目目录。
5. 将原始数据放入本地 `data/raw/` 和 HDFS `/bigdatapj/raw/`。
6. 将成员 3 生成的清洗数据上传到 HDFS `/bigdatapj/cleaned/`。

### 预期效果

1. Hadoop 集群可以正常启动。
2. HDFS 可以正常读写。
3. YARN 可以调度 MapReduce 任务。
4. 成员 5 可以直接在 Hadoop 上运行异常检测任务。

### 需要修改的文件

| 文件 | 作用 |
| --- | --- |
| `docs/环境部署.md` | 记录 master/worker 的安装、配置、启动、问题处理和成功标志。 |
| `scripts/deploy/check_env.ps1` | 环境检查脚本，检查 Python、Hadoop、HDFS 路径是否可用。后续可补 Linux `.sh` 版。 |
| `scripts/hadoop/init_hdfs.ps1` | 创建 HDFS 项目目录，例如 `/bigdatapj/raw`、`/bigdatapj/cleaned`、`/bigdatapj/results`。 |
| `scripts/hadoop/upload_cleaned.ps1` | 把本地清洗数据上传到 HDFS，供 Hadoop Streaming 使用。 |
| `scripts/import_data/README.md` | 记录原始数据导入方式和交付给成员 3 的字段要求。 |
| `data/raw/` | 放小样本或导入后的原始指标数据。真正大数据不要提交 Git。 |

### 需要交付的结果

```text
Hadoop01: NameNode、SecondaryNameNode、ResourceManager
Hadoop02: DataNode、NodeManager
Hadoop03: DataNode、NodeManager
/bigdatapj/raw
/bigdatapj/cleaned
/bigdatapj/results/profiles
/bigdatapj/results/anomalies
/bigdatapj/results/performance
data/raw/metrics_raw.csv 或 HDFS 原始数据路径说明
```

### 验收标准

```text
1. Hadoop01 执行 jps 能看到 NameNode、SecondaryNameNode、ResourceManager。
2. Hadoop02/Hadoop03 执行 jps 能看到 DataNode、NodeManager。
3. hdfs dfs -ls /bigdatapj 能成功。
4. hadoop-mapreduce-examples pi 任务能 completed successfully。
5. NameNode Web 页面 Live Nodes = 2。
6. YARN Web 页面能看到两个 worker 节点。
```

## 3. 成员 3：数据清洗 / 时间序列构造 / 统计画像负责人

### 需要实现的功能

1. 读取成员 2 提供的原始性能指标数据。
2. 统一字段为 `timestamp, cmdb_id, kpi_name, value`。
3. 将 `timestamp` 转成可排序时间格式。
4. 将 `value` 转成数值。
5. 删除空值、重复值、非法值。
6. 按 `cmdb_id + kpi_name + timestamp` 排序。
7. 构造时间序列索引。
8. 输出数据概览、KPI 分布、序列画像、缺失率统计。
9. 准备小样本和中等样本，方便成员 4、5、6 调试。

### 预期效果

1. 算法组可以直接读取 `metrics_cleaned.csv`，不再关心原始数据脏格式。
2. Demo 组可以直接读取 `results/profiles/` 下的统计文件画图。
3. PPT 可以使用统计文件说明数据规模、时间范围、KPI 数量和序列数量。

### 需要修改的文件

| 文件 | 作用 |
| --- | --- |
| `src/preprocessing/clean_metrics.py` | 核心清洗脚本，把原始数据转成标准性能指标接口。 |
| `src/preprocessing/build_profiles.py` | 统计画像脚本，生成数据概览、KPI 分布、序列索引和序列画像。 |
| `src/utils/io.py` | 如遇到新的文件格式，可以在这里扩展读取逻辑。 |
| `data/cleaned/` | 输出清洗后的数据。 |
| `data/sample/` | 输出 small/medium 调试样本。 |
| `results/profiles/` | 输出 Demo 和 PPT 使用的统计文件。 |

### 需要交付的结果

```text
data/cleaned/metrics_cleaned.csv
data/sample/sample_small.csv
data/sample/sample_medium.csv
results/profiles/data_overview.json
results/profiles/kpi_distribution.csv
results/profiles/series_index.csv
results/profiles/series_profile.csv
results/profiles/missing_topn.csv
```

### 输出字段要求

`metrics_cleaned.csv`：

```text
timestamp, cmdb_id, kpi_name, value
```

`series_index.csv`：

```text
series_id, cmdb_id, kpi_name, start_time, end_time, point_count
```

`series_profile.csv`：

```text
series_id, cmdb_id, kpi_name, count, missing_rate, min, max, mean, std
```

### 验收标准

```text
1. metrics_cleaned.csv 字段正确，value 全部可转成数值。
2. 每条序列都能由 cmdb_id + kpi_name 唯一标识。
3. results/profiles 下 5 个统计文件齐全。
4. 成员 4 能用 metrics_cleaned.csv 跑 baseline。
5. 成员 6 能用 data_overview.json 和 kpi_distribution.csv 做页面。
```

## 4. 成员 4：本地 Baseline 异常检测负责人

### 需要实现的功能

1. 按 `cmdb_id + kpi_name` 对清洗数据分组。
2. 实现 IQR 异常检测。
3. 实现 K-Sigma 异常检测。
4. 实现 Range 上下界异常检测。
5. 输出统一格式的异常检测结果。
6. 统计每种算法检测出的异常点数量。
7. 挑选适合 Demo 展示的典型异常序列。

### 预期效果

1. 项目有稳定可解释的保底算法。
2. 即使 Hadoop 高级部分暂时出问题，Demo 也可以展示异常点。
3. PPT 可以讲清楚三种 baseline 算法的原理和效果。

### 需要修改的文件

| 文件 | 作用 |
| --- | --- |
| `src/algorithms/baseline.py` | 本地 baseline 算法入口，包含 IQR、K-Sigma、Range。 |
| `src/utils/data_contract.py` | 只在需要新增结果字段时由组长确认后修改。 |
| `results/anomalies/` | 输出 baseline 异常检测结果。 |

### 需要交付的结果

```text
results/anomalies/anomaly_iqr.csv
results/anomalies/anomaly_ksigma.csv
results/anomalies/anomaly_range.csv
results/anomalies/baseline_summary.csv
results/anomalies/demo_cases.csv
```

### 输出字段要求

所有异常结果文件必须使用：

```text
timestamp, cmdb_id, kpi_name, value, method, is_anomaly, score, threshold_low, threshold_high
```

### 验收标准

```text
1. 三种 baseline 算法都可以独立运行。
2. 输出文件字段和统一异常结果接口一致。
3. 每种算法能统计异常点数量。
4. 能解释 IQR、K-Sigma、Range 的阈值含义。
5. 能提供 3 到 5 条 Demo 典型异常序列。
```

## 5. 成员 5：Hadoop 分布式异常检测 / 高级算法 / 性能测试负责人

### 需要实现的功能

1. 将清洗数据上传到 HDFS。
2. 使用 Hadoop Streaming 运行 Python Mapper 和 Reducer。
3. Mapper 将每行数据转换成 `cmdb_id|kpi_name` 为 key 的键值对。
4. Reducer 对同一条时间序列执行异常检测。
5. 首先跑通 Hadoop-IQR。
6. 扩展滑动窗口 K-Sigma 或 LOF 等高级算法。
7. 输出 Hadoop 检测结果。
8. 做本地单机 baseline 和 Hadoop 分布式任务的运行时间对比。

### 预期效果

1. 能证明项目不是单机脚本，而是使用 Hadoop 完成分布式计算。
2. HDFS 输出目录出现 `_SUCCESS`。
3. `part-*` 文件中包含标准异常结果。
4. PPT 可以展示 Hadoop 作业截图、运行时间和吞吐量。

### 需要修改的文件

| 文件 | 作用 |
| --- | --- |
| `src/hadoop/mapper_baseline.py` | Hadoop Streaming Mapper，负责把 CSV 行转成按序列分组的 key-value。 |
| `src/hadoop/reducer_baseline.py` | Hadoop Streaming Reducer，负责对每条时间序列计算异常结果。 |
| `scripts/hadoop/run_streaming_iqr.ps1` | 提交 Hadoop Streaming IQR 作业的脚本。 |
| `scripts/hadoop/download_results.ps1` | 从 HDFS 下载 `part-*` 结果到本地 `results/`。 |
| `scripts/hadoop/upload_cleaned.ps1` | 把 `metrics_cleaned.csv` 上传到 HDFS。 |
| `src/distributed/shard_runner.py` | 本地模拟分片和 worker 分配，用于性能对比或备用方案。 |
| `src/algorithms/advanced.py` | 高级算法预留文件，需要成员 5 新增。 |
| `results/performance/` | 输出性能测试结果。 |

### 需要交付的结果

```text
/bigdatapj/results/anomalies/hadoop_iqr/_SUCCESS
/bigdatapj/results/anomalies/hadoop_iqr/part-*
results/anomalies/hadoop_iqr/
results/anomalies/anomaly_advanced.csv
results/anomalies/anomaly_events.csv
results/performance/performance_report.csv
```

### 性能结果字段要求

```text
method, server_num, series_count, data_count, runtime_sec, throughput
```

### 验收标准

```text
1. Hadoop Streaming IQR 能成功运行。
2. HDFS 输出目录有 _SUCCESS。
3. part-* 文件中有 timestamp、cmdb_id、kpi_name、value、method、is_anomaly 等字段。
4. 能说明 Mapper、Shuffle、Reducer 分别做了什么。
5. performance_report.csv 能体现单机和 Hadoop 的运行时间对比。
6. 至少完成一个高级或改进算法方案。
```
