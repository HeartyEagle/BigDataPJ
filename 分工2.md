# 成员 4：Baseline 异常检测算法负责人

## 核心定位

成员 4 是项目的“保底结果负责人”。

这个人要保证即使高级算法没完全做完，项目也一定有完整的异常检测结果。

建议成员 4 负责三个基础算法：

```text
IQR
K-Sigma
Range
```

这三个算法容易解释、容易展示、容易和结果图结合。

------

## 具体工作

### 1. 实现 IQR 异常检测

适合检测离群点。

基本思路：

```text
计算 Q1 和 Q3
IQR = Q3 - Q1
下界 = Q1 - 1.5 * IQR
上界 = Q3 + 1.5 * IQR
超过上下界的点认为是异常
```

适合展示的指标：

```text
响应时间
网络流量
磁盘 IO
部分波动型指标
```

------

### 2. 实现 K-Sigma 异常检测

适合检测偏离均值较大的点。

基本思路：

```text
计算均值 mean
计算标准差 std
如果 value > mean + k * std 或 value < mean - k * std
则认为是异常
```

建议初始参数：

```text
k = 3
```

如果用滑动窗口版本，可以设置：

```text
window_size = 60
```

表示用过去 60 个点作为窗口。如果采样频率是 60 秒，那么 60 个点大约是 1 小时。

------

### 3. 实现 Range 异常检测

适合有明确范围的指标。

例如：

```text
CPU 使用率：0 到 100
内存使用率：0 到 100
成功率：0 到 1 或 0 到 100
错误率：通常不应小于 0
```

Range 的好处是特别容易解释，答辩时老师也容易理解。

------

### 4. 输出统一异常结果

成员 4 的输出必须严格按照成员 1 定义的格式。

```text
timestamp, cmdb_id, kpi_name, value, method, is_anomaly, score, threshold_low, threshold_high
```

示例：

```text
2026-01-01 12:00:00, server_001, cpu_usage, 97.5, Range, 1, 97.5, 0, 95
2026-01-01 13:05:00, server_002, response_time, 3500, IQR, 1, 3500, 10, 1200
```

# 成员 5：高级算法 / 并行计算 / 性能测试负责人

## 核心定位

成员 5 负责项目的“加分项”。

成员 4 负责保底，成员 5 负责让项目更像一个“大数据分析技术”项目，而不只是普通 Python 脚本。

成员 5 的工作重点有三个：

```text
高级异常检测方法
分布式/并行运行
性能测试与对比
```

------

## 具体工作

### 1. 实现一个高级算法

建议在下面两个方向里选一个，不要贪多。

#### 方案 A：LOF 多维异常检测

适合多个指标联合判断。

比如同一台机器在同一时间有这些指标：

```text
cpu_usage
memory_usage
disk_io
network_in
network_out
```

单看某个指标可能不明显，但组合起来可能异常。

优点：

```text
比单指标算法高级
适合讲“多维指标关联”
Demo 效果好
```

缺点：

```text
数据整理稍微复杂
需要把多个 kpi 按时间对齐
```

------

#### 方案 B：滑动窗口异常分数

适合时间序列场景。

基本思路：

```text
对每条序列取滑动窗口
计算窗口内均值、标准差、变化率
判断当前点是否偏离历史窗口
```

优点：

```text
贴合时间序列
比全局均值更合理
实现难度适中
```

缺点：

```text
参数需要调，例如窗口大小
```

如果时间有限，建议优先选：

```text
滑动窗口 K-Sigma
```

它比普通 K-Sigma 更像时间序列算法，而且实现成本不高。

------

### 2. 负责并行检测

成员 5 要把“5000+ 序列”拆分到服务器 2 和服务器 3 上跑。

可以按序列 ID 分片：

```text
服务器 2：series_id 取偶数
服务器 3：series_id 取奇数
```

或者：

```text
服务器 2：node/container 类指标
服务器 3：jvm/istio/service 类指标
```

最终要能说清楚：

```text
我们将 5000+ 条时间序列按 series_id 进行分片，
分配到两个计算节点并行执行异常检测，
主节点负责汇总检测结果。
```

------

### 3. 做性能测试

这是很好的加分点。

至少做这几组对比：

| 实验               | 说明                          |
| ------------------ | ----------------------------- |
| 单服务器运行       | 所有序列在一台服务器上跑      |
| 双 Worker 并行运行 | 服务器 2 和服务器 3 分片并行  |
| 不同算法耗时对比   | IQR、K-Sigma、Range、高级算法 |
| 不同数据规模耗时   | 100 条、1000 条、5000+ 条序列 |

输出文件：

```text
performance_report.csv
```

格式：

```text
method, server_num, series_count, runtime_sec, throughput
```

示例：

```text
IQR, 1, 5000, 120.5, 41.49
IQR, 2, 5000, 68.3, 73.21
K-Sigma, 1, 5000, 150.2, 33.29
K-Sigma, 2, 5000, 82.7, 60.46
```

------

### 4. 异常事件聚合

这个可以作为小加分项。

不要只输出孤立异常点，可以把连续异常点合并成异常事件。

例如原始异常点：

```text
12:01 异常
12:02 异常
12:03 异常
12:04 异常
```

合并成：

```text
异常事件 1：12:01 - 12:04，持续 4 分钟
```

输出：

```text
anomaly_events.csv
```

格式：

```text
event_id, cmdb_id, kpi_name, method, start_time, end_time, duration, anomaly_count, max_score
```

这个很适合 Demo 展示 Top-N 异常事件。

# 第二版方案的核心优势

这个分工比第一版更合理，原因是：

```text
1. 数据统计没有单独占一个人，避免人力浪费。
2. Demo 同学可以展示统计图，但不用自己处理原始大数据。
3. 数据清洗同学顺手产出统计文件，工作链路更自然。
4. 算法组有清晰输入，不会被数据格式卡住。
5. 三台服务器分工明确，能体现集群实验环境。
6. 最后 PPT、Demo、文档都有明确负责人。
```

------

# 最终可以这样向组员解释

你可以直接这样说：

> 我们第二版分工不单独设数据分析岗。数据统计这部分由数据清洗同学在预处理时顺手产出，比如序列数量、KPI 分布、缺失率、均值方差等；Demo 同学负责把这些统计结果做成图表，放在系统首页和 PPT 数据集介绍里。这样既保留了数据分析展示，又不会浪费一个完整人力。整体上我们按“环境数据组、算法组、展示集成组”来推进：2 号负责环境和数据导入，3 号负责清洗和统计产出，4 号负责基础算法，5 号负责高级算法和并行性能，6 号负责 Demo 和文档，1 号负责架构集成和答辩总控。

我建议你们就按这个第二版执行。关键是**先统一数据格式和异常结果格式**，然后每个人围绕自己的交付物推进，最后集成会轻松很多。
